# Index raw K8s manifest folders (no kustomize build)

**Type:** AFK
**Category:** enhancement
**State:** ready-for-agent

## What to build

Add support for indexing **raw manifest folders** — directory trees of hand-written, checked-in Kubernetes YAML (`*.yaml`/`*.yml`) read directly, with no `kustomization.yaml` and no `kustomize build`. Raw folders are declared in a new `k8s_manifests` config key, separate from `k8s_overlays` (see ADR 0001). Both source subtypes feed the *same* parse → upsert pipeline, so the resulting graph is indistinguishable by source — only ingestion differs.

End-to-end behavior:

- A new raw reader globs a directory **recursively** for `*.yaml`/`*.yml`, splits multi-document files, and returns resource docs in **deterministic (sorted) order**. Files named `kustomization.yaml`/`.yml`/`Kustomization` are skipped (they are build directives, not resources).
- `index_k8s` indexes `k8s_manifests` paths in a second loop after overlays, reusing the existing overlay glob-resolution logic (`*`/`?`/`[…]` expand to directories under the repo root; "matched no directories" warning preserved). The upsert-nodes-then-edges core is shared between overlay and manifest indexing.
- `env` is derived from the last path component of each configured/expanded path, identical to overlays (`manifests/prod` → `prod`).
- **Resilience:** a malformed YAML file is reported as a warning and skipped; indexing continues over the remaining files and the run still exits 0.
- **Advisory:** if a `kustomization.yaml` is found inside a `k8s_manifests` path, emit a one-line warning per directory suggesting the path may belong in `k8s_overlays`.
- `k8s_manifests` defaults to `[]` when absent from `.graphsearch.toml`, so existing configs keep working. `graphsearch init` prompts for raw manifest paths; the `graphsearch index` "nothing to index" guard accounts for `k8s_manifests`.

## Acceptance criteria

- [ ] `GraphsearchConfig` has a `k8s_manifests: list[str]` field; `load_config` defaults it to `[]` when the key is absent; `write_config` round-trips it.
- [ ] `graphsearch init` prompts for raw manifest paths and records them; `cmd_index` runs indexing when only `k8s_manifests` is configured.
- [ ] Raw reader recursively reads `*.yaml`/`*.yml`, splits multi-doc files, skips empty/`None` docs, ignores other extensions, and returns docs in sorted order.
- [ ] `kustomization.yaml`/`.yml`/`Kustomization` files are skipped, and a per-directory advisory warning is emitted when one is present.
- [ ] A malformed YAML file produces a warning and is skipped; other files in the tree are still indexed; the run exits 0.
- [ ] `k8s_manifests` glob entries (e.g. `manifests/*`) expand to one env per directory, identical to `k8s_overlays`; `env` is the last path component.
- [ ] Overlay and manifest indexing share a single upsert core; docs from the raw reader produce the same nodes/edges as the same docs from kustomize render.
- [ ] README documents `k8s_manifests` and the kustomize-vs-raw distinction.
- [ ] Tests cover: config round-trip + default; recursive/multi-doc/extension-filtering reads; sorted order; kustomization skip + advisory; malformed-skip + exit 0; glob env-tagging; render-vs-raw graph parity.

## Out of scope

- `.json` manifests (YAGNI).
- Auto-detect or typed config entries (rejected in ADR 0001).
- Single-file (non-directory) `k8s_manifests` entries; globs expand to directories only.

## Blocked by

None - can start immediately.

**Category:** enhancement
**Summary:** Index folders of hand-written K8s YAML directly (no `kustomize build`), declared in a new `k8s_manifests` config key.

**Current behavior:**
K8s indexing only consumes `k8s_overlays` — each entry is a directory with a `kustomization.yaml` that is rendered via `kustomize build` (with `--enable-helm` when `helmCharts` is present), and the rendered multi-doc stream is parsed into nodes/edges and upserted. A directory without a `kustomization.yaml` cannot be indexed — render-mode detection raises when no kustomization file is found.

**Desired behavior:**
A second source subtype, the **raw manifest folder**, is read directly with no subprocess. Configured paths resolve and tag `env` identically to overlays (glob expansion to directories under the repo root; `env` = last path component). For each resolved directory the reader walks the tree recursively, reads every `*.yaml`/`*.yml`, splits multi-document files, skips empty docs, and returns the resource docs in deterministic sorted order. Files named `kustomization.yaml`/`.yml`/`Kustomization` are not treated as resources; if one is present in a raw path, emit a one-line advisory (per directory) that the path may belong in `k8s_overlays`. A file that fails to parse is reported as a warning and skipped, and indexing continues over the rest with a successful (zero) exit. The docs produced flow through the exact same parsing and upsert logic as rendered overlays, yielding an identical graph shape.

**Key interfaces:**
- The parsed config type — gains a `k8s_manifests` string-list field, defaulting to empty when the key is absent (existing configs unaffected). The config reader/writer round-trip it, and `init` prompts for it.
- The raw reader — a new unit that takes a directory path and returns a list of resource-doc dicts (the same shape kustomize rendering yields), plus signalling for skipped/malformed files and detected kustomization files.
- The K8s index orchestrator — indexes `k8s_manifests` paths alongside overlays, sharing a single "upsert nodes then edges" core with the overlay path; the parse/upsert stage must be source-agnostic.
- The `index` command's "nothing to index" guard — must account for `k8s_manifests`.
